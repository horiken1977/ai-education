# 設計の整合を保つ仕組み（メンテ規約）

> 目的：台本・スライド・設計書・ナビが**バラバラに更新されてズレる**事故を防ぐ。
> 過去にカリキュラム設計書が2本（旧/改訂版）に分裂して二重持ちになり、台本だけ更新されて設計書が取り残された。その再発防止。

## 1. 正本マップ（Single Source of Truth）
「ここだけが正」を1か所に決め、他はそれに**従う／リンクする**（同じ内容を二重に書かない）。

| 対象 | 正本（ここだけが正） | 追従する場所 |
|---|---|---|
| レベル定義・卒業条件・スケジュール・ゴール分布・業務フロー | **`design/curriculum.md`** | `scripts/*`・`slides/*`・`mkdocs.yml` nav・レベル診断Forms |
| オンボで受講者に説明する正本ブロック（①前提＝営業プロセス変革／②設計の重心／③レベル定義と卒業条件／④学びの変化＝失敗して修正／⑤3ヶ月スケジュール・混在型） | **`design/curriculum.md` 冒頭ほか** | **`scripts/onboarding.md`（台本＝話し言葉）と `slides/onboarding.md`（スライド①〜⑤＝縮約版）の両方に必須**（§3-5の番人が監視）。※背骨（業務フロー×渡し方）はオンボ対象外＝LV1以降＋設計書§3で扱う |
| 各レベルの分刻み進行（セリフ・操作・チャット） | **`scripts/<level>.md`（台本）**。※**オンボーディングは `slides/onboarding.md`（スライド）が正＝台本が追従**（`/slide-sync` で反映） | `slides/<level>.md`（オンボ以外は台本の要点を縮約） |
| 安全・録画・同意の共通ルール | **`scripts/index.md`（台本ガイド）** | 各台本は参照（再掲は最小限） |
| GitHub運用・ビルド・公開 | **`design/github-ops.md` ＋ `mkdocs.yml` ＋ `deploy.yml`** | — |
| 講義モード（Marp）方針 | **`design/lecture-mode.md`** | — |
| ダッシュボード | **`dashboard/` ＋ `design/curriculum.md` §8** | — |

原則：**設計書は「定義とアウトライン」まで。分刻みの中身は台本が正**（重複させない＝ズレない）。

> **レベル定義の表は自動同期（単一の正）**：`curriculum.md` の `--8<-- [start:levels]`〜`[end:levels]` がマスタ。ホーム等は `--8<-- "design/curriculum.md:levels"` で取り込むので、**マスタ表を1か所直せばWebも自動更新**される。
> （nav のラベル・台本/スライド本文・ダッシュボードのJS配列は手動 → §2のチェックリスト＋CIの旧表記grepで担保。）

## 2. 変更時チェックリスト
### レベルを追加・改名・再定義したら
- [ ] `design/curriculum.md`（§2 定義／§3 業務フロー／§4 スケジュール／§6 アウトライン）
- [ ] 該当 `scripts/<level>.md`（台本）
- [ ] 該当 `slides/<level>.md`（スライド）
- [ ] `mkdocs.yml` nav（講師台本・配布教材・の各ラベル/パス）
- [ ] `docs/index.md`（トップのレベル表・スライドリンク）
- [ ] `scripts/index.md`（台本ガイドのリンク行）
- [ ] レベル診断Forms（出る回を選ぶ5問）
- [ ] ダッシュボードの分布ラベル（§8）

### 受講者向け正本ブロック（前提・重心・背骨・レベル）を足す/直したら
- [ ] `design/curriculum.md` 冒頭（正本）を更新
- [ ] `scripts/onboarding.md`（台本）に**話し言葉のビート**で反映
- [ ] `slides/onboarding.md`（スライド）に**縮約版スライド**で反映
- [ ] `tools/check_onboarding_sync.py` の `CHECKS` に**キー概念マーカー**を追加（揺れにくい語を選ぶ）→ ローカルで `python3 tools/check_onboarding_sync.py` が緑

### ドキュメントを削除・改名したら
- [ ] `grep -rn "<旧ファイル名>" docs mkdocs.yml` で参照ゼロを確認し、全て正本へ付け替え

> 迷ったら **正本（`curriculum.md`）に合わせる**。

## 3. 自動整合チェック（CI）
`.github/workflows/consistency.yml` が push / PR ごとに自動実行：
1. **`mkdocs build --strict`** … リンク切れ・ナビ未登録・参照ミスを**エラーで検出**（構造の整合）。
2. **廃止ファイル参照の検査** … `curriculum-v2` / `dashboard-m365` 等の削除済みファイルへの.mdリンクが残っていたら**失敗**。
3. **旧表記の検査** … 旧レベル名・旧プログラム名（例：`L1 初心者` / `営業向けAI研修`）が docs に残っていたら**失敗**。
4. **重複・ダブりの検査**（`tools/check_dup.py`）… ①レベル定義表がマスタ以外に直書き＝**データ重複** ②同一ページ内の見出し(H1/H2)重複＝**表示ダブり** ③nav の同一パス重複 を**失敗**で検出。
5. **受講者向け正本の整合**（`tools/check_onboarding_sync.py`）… 設計書冒頭の受講者向けブロック（前提・重心・背骨・レベル）の**キー概念が、台本(`scripts/onboarding.md`)とスライド(`slides/onboarding.md`)の両方に在るか**を検査。片方に入れ忘れたら**失敗**（台本とスライドは別ビルドでズレやすいため）。

赤（失敗）が出たら整合が崩れているサイン。**公開デプロイ（`deploy.yml`）とは別ワークフロー**なので、チェックが赤でもサイトは更新される（＝チェックは「気づくための番人」）。

### ローカルでの確認
```bash
bash tools/check_slides.sh   # 速い整合チェック一括（重複＋受講者向け整合＋旧表記／mkdocsがあればstrictも）
# （フル確認したい時）
pip install -r requirements.txt
mkdocs build --strict        # 警告＝エラーで止まる。何も出なければ整合OK
```
> **スライドを手編集したら**：`tools/check_slides.sh` を緑にしてから push。設計書への逆反映・台本への反映・CI・公開まで一気にやるなら **スキル `/slide-sync`**（`.claude/commands/slide-sync.md`）を使う。編集時の「消さない語」は `slides/onboarding.md` 冒頭の非表示コメントにも明記。

> セマンティックな中身（台本と設計書の“言っていること”の一致）は完全には自動化できない。ただし**受講者向け正本ブロックだけはキー概念マーカーで台本⇄スライドの取りこぼしを機械検出**（§3-5）。残りは**§1の正本マップ＋§2のチェックリスト**で人が担保し、CIは**構造のズレ**を機械で止める、という二段構え。
